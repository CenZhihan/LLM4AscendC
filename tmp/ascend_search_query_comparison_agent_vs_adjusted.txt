================================================================================
Ascend 在线文档检索：Agent 当时 ascend_search 的 query vs 调整后更易命中的 query
================================================================================

说明
----
- Agent 侧 query 来源：`output/test/ascendc/gpt-5/agent_ascend_fetch_ascend_search/one_shot/run0/` 下
  `gelu_report.json` / `elu_report.json` / `softmax_report.json` 的 `tool_calls` 中 `tool=="ascend_search"` 的 `query` 字段。
- 「调整后」的 query：与仓库 `ascend_search_node` 一致策略（`lang=zh`、`doc_type=DOC`、`version_filter=8.5.0`）
  下，人工收紧关键词后，用 `AscendDocsSearchRetriever` 实测更容易首条命中「能对照改错 / 补工程」文档的写法。
- 本文档为对比备忘，不修改线上 prompt；若写入 agent 的 choose_tool 提示，请再经团队评审。

--------------------------------------------------------------------------------
一、GELU（编译问题：内核里 Tanh 重载/第三参临时空间类型与 CANN 头文件不一致）
--------------------------------------------------------------------------------

| 轮次 | Agent 当时 ascend_search.query | 主要问题（为何易引入误差） |
|------|----------------------------------|----------------------------|
| 1 | AscendC tanh erf 函数 接口 文档 | **tanh+erf+「文档」** 混在一query，首条易成 **HCCL AlltoAllV** 等与 GELU 无关条目，噪声大。 |
| 2 | AscendC Erf 接口 | 合理查 Erf，但与 **Tanh 临时空间/重载** 目标偏离；易把后续 fetch 引向 Erf 辅助页。 |
| 3 | AscendC Tanh 接口 | **较好**：直接命中 Tanh 数学 API（07_0492）。 |
| 4 | AscendC tanh 向量 函数 API | 与轮次3信息重复度高，仍围绕 Tanh，尚可。 |

建议的「调整后」query（优先顺序示例）
  1) AscendC Tanh 接口 临时空间
     → 首条常为 Tanh 接口页，正文强调 sharedTmpBuffer / uint8 / LocalTensor，与「第三参为临时缓冲」强相关。
  2) GetTanhMaxMinTmpSize host
     → 首条为 GetTanhMaxMinTmpSize（07_0493），补 **host 侧取临时空间大小** 与 kernel 衔接，减少只抄 Tanh 四参形式句而忽略缓冲语义。

对比小结
  Agent 前几轮 **过宽或目标分裂**（tanh+erf+文档），**第3轮起才稳定到 Tanh**；若首轮即用「Tanh + 临时空间」并补一条 **GetTanhMaxMinTmpSize host**，可更早收敛到与编译错误直接相关的 API 组合。

--------------------------------------------------------------------------------
二、ELU（编译问题：op_host 中 AscendC::GetExpMaxMinTmpSize 未声明 / host 工程与头文件语境）
--------------------------------------------------------------------------------

| 轮次 | Agent 当时 ascend_search.query | 主要问题 |
|------|----------------------------------|----------|
| 1 | AscendC Exp 接口 | **好**：命中 Exp（07_0589）与 GetExpMaxMinTmpSize 线索，与 ELU 数学路径一致。 |
| 3 | AscendC Mins 接口 文档 | 为 ELU 分支需要，但与 **host 侧 GetExpMaxMinTmpSize 声明/头文件** 关系弱。 |
| 4 | AscendC Maxs 接口 文档 | 同上，偏 **kernel 矢量基础 API**，对「op_host 编译不过」帮助间接。 |

建议的「调整后」query（分步，避免首条跑偏）
  1) GetExpMaxMinTmpSize Exp 接口
     → 先锁 **API 名 + Exp 父主题**，避免「host 头文件」等词把排序拽到 **故障处理/stackcore** 类文档。
  2) 自定义算子 tiling 头文件
     → 首条可落到 **Host 侧 Tiling 实现 / 附录**（如 atlas_ascendc_10_00025），正文常含 **TilingFunc、#include、头文件**，补 **工程化与 include 组织**。
  （若仍要搜头文件）勿与冷门词强绑首条：可再试「Host侧Tiling实现 附录」等。

对比小结
  Agent **Exp 首轮正确**，但后续轮次多在 **Mins/Maxs** 上扩展，**未专门用 query 把「GetExpMaxMinTmpSize + host/头文件/附录」推到排序前列**；调整后 **拆成「API 名」+「Host Tiling 工程附录」** 更贴编译失败点。

--------------------------------------------------------------------------------
三、Softmax（编译问题：host TilingFunc 里误把 GetInputShape 当 gert::Shape* / GetOriginShape 用法错误）
--------------------------------------------------------------------------------

| 轮次 | Agent 当时 ascend_search.query | 主要问题 |
|------|----------------------------------|----------|
| 1 | AscendC ReduceMax 接口 文档 | **偏题**：首条易为 **AdjustSoftMaxRes**（SoftMax 后处理），与 **TilingContext 取 shape 类型** 无关。 |
| 3 | SoftMax 接口 AscendC 高阶API | **对 kernel 侧 SoftMax API 好**，但几乎 **不覆盖 gert::TilingContext::GetInputShape 返回 StorageShape*** 的 host 说明。 |

建议的「调整后」query（针对 gert 基础 API）
  1) TilingContext 获取输入 shape
     → 首条常为 **GetInputShape - TilingContext**（atlasopapi_07_00224），正文含原型：
     `const StorageShape *GetInputShape(const size_t index) const` 及 StorageShape 说明。
  2) StorageShape GetOriginShape tiling
     → 首条常为 **GetOriginShape - StorageShape**（atlasopapi_07_00176），与从 StorageShape 取原始/运行时 shape 衔接。

对比小结
  Agent 用 **ReduceMax** 开场，**排序先落到 SoftMax 周边高阶 API**，**未命中 gert 基础数据结构章节**；对「StorageShape* vs Shape*」这类错误，**必须在 query 中显式出现 TilingContext / StorageShape / GetInputShape** 之一。

--------------------------------------------------------------------------------
五、检索策略提醒（与线上 ascend_search_node 对齐）
--------------------------------------------------------------------------------

- 线上一律：`doc_type=DOC`、`version_filter=8.5.0`（见 `generator/agent/nodes/ascend_search.py`）。
- `query` 必须含 **中文**（节点会拒绝纯英文）。
- 目标越「底层类型/API 名」，query 越应 **显式写出 API 符号**（GetInputShape、StorageShape、GetExpMaxMinTmpSize），
  少用仅语义大类词（ReduceMax、接口 文档）作首轮，以免首条被相邻域文档占住。
