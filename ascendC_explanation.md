## Ascend C 算子：通用流程与产物说明

这份文档面向读者，描述 **开发流程**、**评测流程**、以及评测中常见的 **产物类型** 与 **修改后的覆盖/复用原则**。不依赖任何具体项目结构或路径。

---

## 1) 算子标准开发流程（从 0 到可用）

可以按 6 个阶段理解：**定义 → Host → Kernel → 构建安装 → 接入 → 验证迭代**。

- **阶段 1：需求与接口定义**
  - 明确数学语义（算子做什么）。
  - 明确输入/输出：数量、shape 约束、dtype、format（例如 ND/NCHW）、是否需要 workspace。
  - 命名约定：算子名（CamelCase）与 kernel 名（snake_case）。

- **阶段 2：Host 侧实现（让算子可被调度）**
  - 注册：声明输入/输出、支持的 dtype/format、目标硬件（SoC）。
  - 推导：InferShape / InferDataType（输出 shape/type 如何从输入得到）。
  - Tiling：决定核数（blockDim）、每核处理范围、tile 大小；将 tiling 数据下发给 kernel；必要时设置 workspace 大小。

- **阶段 3：Kernel 侧实现（真正的计算）**
  - 根据 tiling 数据确定当前核处理范围。
  - 进行数据搬运 / 计算 / 回写，并处理尾块与边界。
  - 常见注意点（只保留概念）：
    - 数据搬运的对齐与长度约束；
    - 多核切分的边界一致性；
    - 同步与流水（避免读写冲突）。

- **阶段 4：构建与安装（产出可加载的算子包）**
  - 生成工程骨架（通常包含：构建脚本、CMake、自动生成代码、打包脚本）。
  - 合入 Host/Kernel 源码后编译生成二进制与打包产物。
  - 执行安装（将算子相关文件安装到运行时可搜索的位置）。

- **阶段 5：调用侧接入（可选，但很常见）**
  - 在框架侧提供调用入口（例如 C++ 调用或 Python 调用）。
  - 若做 Python 调用，通常需要编译一个扩展模块（.so / wheel）封装调用与参数检查。

- **阶段 6：验证与迭代**
  - 先 correctness（正确性），再 performance（性能）。
  - 最小化变量：先用小 shape、对齐友好、固定切分跑通；再逐步打开多核切分与性能优化。

## 2) 算子标准评测流程（应经历哪些步骤）

建议把评测拆成两段：**Build/Deploy** 与 **Eval**。

- **(A) Build/Deploy（生成、编译、部署）**
  - 环境准备（工具链、运行时库路径、Python 环境等）。
  - 生成工程骨架。
  - 合入 Host/Kernel 实现代码。
  - 编译生成目标产物（包含 kernel 二进制、Host 侧库等）。
  - 安装到运行时可加载的位置。
  -（可选）构建并安装 Python 扩展模块/绑定，用于评测端调用。

- **(B) Eval（运行与对比）**
  - 构造测试输入（随机/固定种子；覆盖典型 shape + 边界 shape）。
  - 调用自定义算子在 NPU 上运行。
  - 用参考实现生成对照结果（通常在 PyTorch/NumPy 里表达同一数学语义）。
  - 误差判定：`allclose`（rtol/atol）+ 最大误差等统计。
  - 记录日志：输入规格、版本信息、误差、是否通过。

---

## 3) 评测会产出哪些中间文件/内容？（按“类型”分类）

评测过程常见 5 类产物：

- **A. 源码（算子本身）**
  - Host 侧源码（注册、推导、tiling）
  - Kernel 侧源码（计算与搬运）
  -（可选）调用侧源码（例如 Python/C++ 绑定）
  -（可选）评测用例源码（输入构造、参考实现、比较逻辑）

- **B. 工程骨架与构建中间产物（建议可随时删）**
  - 自动生成的工程文件（构建脚本、CMake、autogen 代码等）
  - 编译中间文件（obj、依赖、临时目录）
  - 打包产物（例如可安装包）

- **C. 安装到运行时的产物（系统可加载内容）**
  - Host 侧库、kernel 二进制、算子描述/配置文件等
  - 这类产物不一定在项目目录内：可能安装到系统/工具链目录

- **D. 调用侧扩展产物（可选）**
  - Python wheel / .so 等
  - 这类产物通常需要“强制重装”以避免旧版本残留

- **E. 评测日志与状态记录（建议保留）**
  - 分步骤日志：生成/编译/安装/运行/对比
  - 状态记录：构建版本号、源码指纹（哈希）、本次构建产物信息等

## 4) 算子修改后：哪些必须覆盖？哪些可以复用？

### 4.1 必须覆盖/重建（为了避免“用旧产物评测”的假象）

只要修改了下列任意一项，都应触发一次完整的 Build/Deploy（并在需要时重装绑定）：

- **Kernel 实现改动**：任何计算、搬运、同步、分块、边界处理改动
- **Host 实现改动**：tiling 策略、推导逻辑、支持的 dtype/format、workspace 等
- **接口改动**：输入/输出数量、dtype、shape 约束、算子名/符号名、绑定导出函数等
- **评测语义改动**：参考实现、rtol/atol、测试 shape 覆盖范围

对应需要覆盖的产物类型通常是：

- 构建中间产物（B 类）
- 安装到运行时的产物（C 类）
- 调用侧扩展产物（D 类，建议 force reinstall）

### 4.2 通常可以复用（不需要随算子改动而频繁修改）

- 通用评测框架/工具脚本（只要算子接口不变）
- 通用日志、执行器、指纹机制（如果有）
- 通用用例模板（“生成输入 → 调用 → 对比 → 输出”的骨架）

### 4.3 不用太改，但仍然是“每个算子都要有一份”的专属内容

- 算子元信息（算子名、输入输出描述、支持平台等）
- 该算子的调用入口/绑定代码（结构可能固定，但绑定到具体算子符号）
- 该算子的评测用例配置（测试 shape/dtype、参考实现、误差阈值）
